# RAYPLAN.md

## Mission

We want the smallest Ray-based distributed trainer that is actually useful:

- one learner process
- one shared policy being optimized by that learner
- multiple rollout actors collecting experience in parallel
- optional separate evaluation actor
- the current PPO + SNN + reward shaping stack kept intact

This is intentionally **not** "rewrite everything into RLlib" and **not** "jump to IMPALA/V-trace immediately". The safest first win is to keep our current PPO semantics and only distribute rollout collection.

---

## 1. What are we actually building?

**Question:** What is the simplest distributed version of our current system?

**Answer:** A **synchronous actor-learner PPO**:

- actors run environments and collect rollout fragments
- the learner waits until it has enough fresh transitions
- the learner performs PPO updates once on that aggregated batch
- then the learner publishes a new weight version
- actors pull that version before collecting the next fragment

This preserves PPO's on-policy assumptions much better than a fully asynchronous setup.

**Why not go fully async on day 1?**

- PPO is happiest when the data was generated by a policy very close to the current learner policy.
- If actors keep collecting with stale weights while the learner keeps updating, we create policy lag.
- Policy lag is exactly where methods like IMPALA + V-trace become useful.

So the first Ray milestone should be:

- **parallel rollout collection**
- **single learner**
- **synchronous update boundary**

That gives us most of the CPU scaling benefit without changing the algorithm class.

---

## 2. What stays where?

**Question:** Which logic belongs on actors, and which logic must stay on the learner?

**Answer:**

### Learner owns

- the master `PolicyNetwork`
- the `PPO` optimizer and scheduler
- checkpoint save/load
- policy versioning
- rollout aggregation
- GAE / return computation
- minibatch PPO updates
- best-model selection

### Rollout actor owns

- one `PySC2` environment
- one inference-only copy of the policy
- one `ObservationExtractor`
- one `RewardFunctionV2`
- one local `snn_state`
- one episode lifecycle (`reset`, `step`, timeout handling)

### Eval actor owns

- a deterministic copy of the policy
- its own environment
- periodic evaluation only

### Logger owns

- one write path to SQLite (or later a safer append-only event sink)

This split matters because the learner should be the **only** thing mutating optimizer state. Actors should never step an optimizer.

---

## 3. What is the biggest conceptual refactor before Ray?

**Question:** Why can't we just spawn N copies of the current trainer and concatenate their memories?

**Answer:** Because the current PPO buffer is still shaped like **one linear rollout**.

Right now `PPO` stores:

- one flat `memory` list
- one flat `final_next` bootstrap tail

That works for the current single-agent trainer, but it is not enough for many actor fragments. If we collect fragments from Actor A, Actor B, and Actor C, there is **not one final bootstrap state**, there is one bootstrap boundary per fragment. (Important limitation: `PPO.final_next` can only represent one non-terminal tail for the whole batch, so naive multi-actor concatenation would compute wrong advantages for every fragment except the last one.)

So the first real refactor is:

- move from "one monolithic memory list" to **`RolloutFragment` objects**
- compute GAE **per fragment**
- concatenate only after returns/advantages are valid

That is the key data-model change that unlocks Ray cleanly.

---

## 4. What should cross the Ray boundary?

**Question:** What data do actors send back to the learner?

**Answer:** Not raw `PySC2` observations. Actors should send **compact rollout fragments** containing only the tensors and metadata the learner needs.

Recommended fragment payload:

- `actor_id`
- `policy_version`
- `fragment_id`
- `num_learnable_steps`
- `episode_count_crossed`
- `terminated` / `truncated`
- `spatial_obs`
- `vector_obs`
- `actions`
- `move_x`
- `move_y`
- `log_probs`
- `rewards`
- `values`
- `dones`
- `pre_step_snn_state`
- `tail_next_spatial`
- `tail_next_vector`
- `tail_next_snn_state`
- `episode summaries`
- `reward component summaries`
- `step counters`

**Why this shape?**

- the learner needs old log-probs and values for PPO
- the learner needs pre-step SNN state for replay
- the learner needs one tail bootstrap per fragment
- the logger needs episode/step/reward metadata without reconstructing it later

**What should not cross the boundary?**

- live `PySC2` env objects
- raw action function objects
- optimizer state
- full learner object

## 5. What code smells already matter for distributed work?

These are not all catastrophic today, but they become real blockers once Ray enters the picture:

- `Utility/config.py` loads `config.yaml` using the current working directory, not an absolute repo-relative path. (Bug/risk: Ray workers often start from a different working directory, so config loading can silently fail or pick up the wrong file.)
- `envs/setup_env.py` hardcodes `screen=84` even though the config now exposes `model.screen_size`. (Bug: model/env geometry can drift if we ever change screen size in config.)
- `envs/setup_env.py` defaults to `"DefeatZerglingsAndBanelings"` while the project config targets `"DefeatRoaches"`. (Bug/risk: a future direct call to `create_env()` without config wiring can spawn the wrong map.)
- `PPO_CNN_run.py` passes `observation_extractor` into `train_agent()` and immediately deletes it. (Bug/smell: this is harmless but tells us the runtime boundary is still muddy.)
- `PPO_CNN_run.py` prints `"Starting Async PPO training..."` even though the current trainer is not async. (Bug/smell: naming drift will get more confusing once we add actual distributed async components.)
- `Utility/logger_utils.py::_safe_add_column()` suppresses every `sqlite3.OperationalError`, not just duplicate-column cases. (Bug: real schema migration failures can be hidden.)
- `maybe_run_policy_update()` logs PPO updates after episode close using `internal_ep = episode_index - 1`, while `LogListener` resolves `episode_id` through a map that is deleted on `EPISODE_END`. (Bug: `ppo_updates.episode_id` can end up `NULL`, and that gets harder to reason about once many actors emit updates.)
- `run_eval_sweep()` reuses the training env object instead of a dedicated eval env. (Bug/risk: acceptable in the single-process loop because the next training episode resets anyway, but not a good pattern for distributed execution.)

## 6. What architecture do I recommend?

**Question:** Should we use RLlib, or write a thin custom Ray layer around our code?

**Answer:** Write a **thin custom Ray layer first**.

Why:

- we already have a custom policy interface
- we already carry explicit SNN state across steps
- we already have custom action semantics and reward shaping
- we already have custom logging/checkpoint behavior

RLlib can still be a later experiment, but the first distributed version should preserve our current internal contracts so we can debug behavior instead of debugging a framework migration.

Recommended process layout for v1:

1. `LearnerCoordinator`
2. `RolloutActor` x N
3. `EvalActor` x 1
4. `LoggerActor` x 1

And recommended dataflow:

1. learner publishes `policy_version = V`
2. actors pull weights V
3. each actor collects `fragment_steps` learnable transitions
4. actors return fragments to learner
5. learner aggregates until `global_rollout_steps` is reached
6. learner computes GAE per fragment
7. learner runs PPO update(s)
8. learner publishes `policy_version = V + 1`
9. eval actor periodically runs deterministic eval on the latest version

## 7. What refactor should happen before the first Ray actor exists?

### Phase 0: Make the single-process code fragment-based

**Question:** What is the smallest local refactor that buys us the most for Ray?

**Answer:** Teach the learner to consume **fragments**, even when they are produced locally.

### Files to touch

- `PPO_CNN/PPO.py`
- `PPO_CNN_agent.py`
- `PPO_CNN_run.py`
- `Utility/logger_utils.py`
- new module: `distributed/protocol.py`

### Concrete changes

1. Introduce explicit transport objects:
   - `TransitionRecord`
   - `RolloutFragment`
   - `EpisodeSummary`
   - `WeightSnapshot`
   - `UpdateSummary`

2. Refactor `PPO.update_policy()` to accept either:
   - a list of fragments, or
   - one already-flattened batch plus fragment boundaries

3. Move GAE computation to run **per fragment**, not once over one global flat list.

4. Stop making `PPO` own the idea of exactly one tail bootstrap.

5. Add a local single-process path that still works without Ray:
   - collect one fragment locally
   - feed it through the same fragment-based learner path

### Why this phase matters

If Phase 0 is skipped, the first Ray prototype will be a pile of serialization code wrapped around a single-process assumption.

### Acceptance criteria

- current single-process training still runs
- PPO update path works from fragments
- existing smoke tests still pass
- one new test proves multi-fragment GAE correctness

## 8. How should the rollout actor behave?

### Phase 1: Build one real `RolloutActor`

**Question:** What should a rollout actor do, exactly?

**Answer:** Own one env and return one fragment generated under one policy version.

### New file candidates

- `distributed/ray_actor.py`
- `distributed/runtime.py`

### Actor API sketch

```python
@ray.remote
class RolloutActor:
    def __init__(self, actor_id, config_path, run_name):
        ...

    def set_weights(self, weights_ref, policy_version):
        ...

    def collect_fragment(self, target_steps, deterministic=False):
        ...

    def get_stats(self):
        ...
```

### Actor rules

- env lives forever inside the actor unless it crashes
- policy replica is inference-only
- policy weights are copied from learner snapshots
- actor keeps its own local `snn_state`, extractor history, and reward state
- actor can cross episode boundaries while collecting a fragment
- actor returns only CPU tensors / NumPy arrays / plain metadata

### Important design choice

Collect a **fixed number of learnable steps per fragment**, not a fixed number of episodes. That preserves the stabilization work we already did in the single-process trainer.

### Acceptance criteria

- one actor can collect a fragment and return it to a local caller
- fragment contains valid bootstrap tail
- fragment can include multiple short episodes
- actor can reset cleanly after terminal episodes

## 9. How should the learner work?

### Phase 2: Build the learner coordinator

**Question:** What stays central, and what timing model should it use?

**Answer:** The learner should own the only mutable optimizer state and update on a **global transition budget**.

### New file candidates

- `distributed/learner.py`
- `distributed/ray_train.py`

### Learner responsibilities

- create the master policy + PPO optimizer
- load checkpoint if resuming
- publish CPU weight snapshots
- dispatch collection requests
- gather fragments
- aggregate until `global_rollout_steps` is met
- update policy
- emit summaries
- checkpoint

### Recommended synchronous training loop

1. publish weights version `V`
2. ask all actors for one fragment
3. gather returned fragments
4. if total transitions < target, request more fragments
5. once target is reached, stop collection
6. compute fragment-local GAE / returns
7. flatten into learner minibatches
8. run PPO update(s)
9. increment policy version

### Why not let actors stream continuously?

Because stale policy data is much harder to reason about, and our current PPO implementation has no off-policy correction layer.

### Acceptance criteria

- 2 actors can feed 1 learner
- learner update count advances exactly once per global batch
- actor fragments all report the same input `policy_version`
- scheduler now keys off **learner update count**, not `total_episodes` guessed inside each agent (bug/risk: `DefeatRoaches.total_updates_estimate` currently assumes one local trainer and becomes meaningless once many actors feed one learner)

## 10. What happens to checkpointing and resume?

### Phase 3: Make resume learner-owned and actor-light

**Question:** What state should a checkpoint contain in a distributed PPO setup?

**Answer:** Learner state, not actor process internals.

### Checkpoint should include

- learner policy weights
- optimizer state
- scheduler state
- global update count
- global environment step count
- best eval reward
- current policy version
- run config snapshot

### Checkpoint should not depend on

- live actor object IDs
- actor-local episode counters
- actor-local env instances

### Practical rule

Actors should be treated as disposable. If one dies, the learner respawns it and sends the latest weights.

### Acceptance criteria

- stop and resume preserves optimizer/scheduler behavior
- actor count can change between resumes without corrupting training
- best checkpoint logic still works

---

## 11. What happens to logging and analysis?

### Phase 4: Replace the local queue/process logging pattern with one distributed writer

**Question:** Why not let every actor write SQLite directly?

**Answer:** Because concurrent multi-process writes from many actors become fragile quickly, and we already have a single-writer shape in spirit.

### New file candidates

- `distributed/logger_actor.py`
- maybe `Utility/log_schema.py`

### Logger changes

- one logger actor owns the SQLite connection
- everyone else emits plain event dicts
- add `actor_id`
- add `policy_version`
- add `fragment_id`
- add `global_update_index`

### DB schema changes worth planning now

- `steps.actor_id`
- `steps.policy_version`
- `episodes.actor_id`
- `ppo_updates.global_update_index`
- `ppo_updates.policy_version`
- `eval_runs.policy_version`

### Why this matters

Without actor/version metadata, distributed debugging becomes guesswork.

### Acceptance criteria

- many actors can emit logs without DB corruption
- analysis can slice by actor and policy version
- update rows are not tied to a fragile episode map anymore (bug: the current episode-map logic is already borderline for single-process update logging)

---

## 12. What about evaluation?

### Phase 5: Dedicated eval actor

**Question:** Should evaluation run inside the learner loop or on one rollout actor?

**Answer:** Use a dedicated `EvalActor`.

Why:

- deterministic eval should not perturb rollout timing
- we do not want evaluation resets mixed into rollout actors
- eval cadence should be tied to learner updates, not to one actor's episode count

### Eval behavior

- receives latest weights and `policy_version`
- runs deterministic episodes only
- logs mean/std/min/max reward
- never contributes data to training

### Acceptance criteria

- eval can lag slightly behind training without harming correctness
- best-checkpoint promotion is tied to eval actor results

## 13. What config changes do we need?

### Phase 6: Make distributed config explicit

Add a new `distributed:` section to `config.yaml`, for example:

```yaml
distributed:
  enabled: false
  num_rollout_actors: 4
  num_eval_actors: 1
  fragment_steps: 256
  global_rollout_steps: 2048
  learner_device: "cuda"
  actor_device: "cpu"
  ray_local_mode: false
  object_store_memory_gb: 8
  actor_cpus: 1
  learner_gpus: 1
```

### Also fix config loading

- load config by absolute path
- pass explicit config path into Ray actors

(Bug/risk: the current config singleton is cwd-relative, which is a footgun for Ray workers and also for mixed Windows/WSL launch paths.)

### Acceptance criteria

- Ray run can be launched from any working directory
- actor count and fragment size are config-driven
- local single-process mode still works from the same config

## 14. What platform decision should we make before implementation?

**Question:** Can we keep the current "Windows for some things, WSL for others" split once Ray enters the picture?

**Answer:** Not for the first distributed version.

For v1, choose **one runtime family** as the source of truth:

- one Python environment
- one filesystem root
- one place where the SC2 binary is known to work
- one place where Ray head + actors + learner all run

Why this matters:

- Ray worker startup is already enough moving parts
- cross-OS path assumptions will multiply confusion
- cwd-relative config and local file outputs are already fragile today

My recommendation:

- first distributed milestone should run on **one machine, one OS/runtime**
- only after that works should we even think about cross-machine or mixed-runtime arrangements

## 15. What tests should we add before calling this "distributed"?

### Phase 7: Test ladder

**Question:** What do we need to prove, in order?

**Answer:** Add tests in layers, from pure data logic to one real Ray smoke.

### Layer A: pure unit tests

- fragment schema round-trip
- multi-fragment GAE correctness
- bootstrap handling for terminal vs truncated fragment tails
- policy-version tagging

### Layer B: local integration tests without Ray

- learner can consume two locally fabricated fragments
- checkpoint/resume with fragment-based PPO still works
- logger can ingest fragment/event payloads

### Layer C: Ray smoke tests

- one learner + one actor
- one learner + two actors
- actor respawn path
- deterministic eval actor path

### Layer D: performance sanity

- measure steps/sec vs current single-process baseline
- measure learner idle time waiting for actors
- measure object payload size

---

## 16. What should we explicitly not do yet?

To keep the first version finishable, do **not** do these in the same PR:

- full RLlib migration
- IMPALA / V-trace
- replay buffer
- multi-node cluster support
- GPU inference on rollout actors
- actor-side gradient computation
- complex fault-tolerant distributed DB writes

Each of those can be a later milestone, but they should not be coupled to the first "1 learner + N actors" version.

---

## 17. What is the recommended build order?

If I were implementing this myself, I would do it in exactly this order:

1. make PPO consume `RolloutFragment` locally
2. move config loading to an explicit path-based API
3. make env creation fully config-driven
4. introduce a learner-owned global update counter
5. introduce a dedicated logging interface that does not depend on episode-map tricks
6. build one local `RolloutActor` class without Ray
7. wrap that actor in Ray and validate one-actor collection
8. add multi-actor synchronous aggregation
9. add dedicated eval actor
10. add checkpoint/resume for distributed mode
11. tune actor count and fragment size

This order matters because it keeps us debugging **one new source of complexity at a time**.

---

## 18. What does "done" look like?

The first distributed milestone is done when all of this is true:

- `ray_train.py` can run 1 learner + 2 rollout actors end-to-end
- the learner consumes fragment-based PPO batches correctly
- deterministic eval still works
- checkpoint/resume works
- logger records actor-aware metrics
- throughput is meaningfully higher than the single-process trainer
- the code still supports the old single-process path as a fallback

If we hit that point, then we can decide whether the next branch is:

- more actors
- better throughput tuning
- RLlib experiment
- or a deliberate move toward IMPALA-style async training

---

## Final recommendation

Start with **custom Ray + synchronous PPO + fragment-based rollouts**.

That path gives us:

- the best reuse of the current code
- the least algorithmic risk
- the clearest debugging story
- and a strong base for later async work if we decide PPO is no longer the right shape

The one architectural change we absolutely should not dodge is the fragment refactor. That is the bridge from "one local memory list" to "many actor rollouts", and everything downstream gets easier once that is done.
